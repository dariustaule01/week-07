Feed Collector Overview and Scope for AI-Capstone-Threat-Intel

The Feed Collector is the collection component of the AI-Capstone-Threat-Intel project. Its role is to gather threat intelligence from public sources, preserve useful source context, and prepare raw findings for later analysis and prioritization. The component does not make final incident response decisions by itself. Instead, it acts as the intake layer for the larger system. A good Feed Collector should be reliable, repeatable, and transparent about where each item came from.

The primary job of the Feed Collector is to monitor public threat intelligence sources such as vendor advisories, government cyber alerts, vulnerability feeds, public malware reports, security research blogs, open-source indicators of compromise, and structured feeds like JSON or CSV threat lists. It should be able to collect from sources with different formats, including web pages, RSS feeds, API endpoints, downloadable text files, and manually curated references. The component should not treat every source as equally reliable. It should preserve the source name, source URL, retrieval time, original title, publication date when available, and any author or organization listed by the source.

The Feed Collector should separate collection from analysis. Collection means retrieving and storing the raw or lightly normalized material. Analysis means judging the importance, severity, or credibility of that material. The collector may perform basic validation, such as checking whether an IP address has a valid format or whether a CVE identifier follows the expected pattern, but deeper interpretation should happen later in the pipeline. This separation makes the system easier to test because collection failures can be diagnosed without confusing them with model reasoning failures.

For the capstone, the Feed Collector should output structured records that downstream components can consume. A record should include the raw text or extracted content, source metadata, extracted entities, collection status, and a stable unique identifier. Useful entities include CVE identifiers, IP addresses, domains, URLs, malware family names, affected products, threat actor names, tactics, techniques, and severity labels. The collector should also record whether a field was directly observed in the source or inferred by the system.

A key design principle is traceability. Every finding must be traceable back to a public source. If the assistant later says that a threat is associated with a specific vulnerability, the user should be able to inspect which feed or report produced that association. Traceability reduces hallucination risk and helps users decide whether they trust the output. The system should avoid losing context during extraction. For example, an IP address alone is less useful than an IP address with the sentence or paragraph explaining why it was mentioned.

The Feed Collector should also support scheduled operation. It may run hourly, daily, or on demand depending on the source. Faster sources, such as vulnerability feeds, may need frequent collection. Slower sources, such as long-form reports, may be collected less often. The collector should avoid unnecessary repeated downloads by storing timestamps, hashes, or source-specific identifiers.

The expected output of the Feed Collector is a clean intake layer for the broader threat intelligence system. It should make later retrieval, summarization, scoring, and alerting easier. Success means that the system can answer questions such as which sources were checked, what new items were collected, which items changed, what entities were extracted, and where each piece of evidence came from. The component should be judged by completeness, source coverage, reliability, traceability, and ease of downstream use.
