Feed Ingestion, Normalization, and Deduplication Procedures

Feed ingestion is the process of retrieving threat intelligence content from public sources and converting it into records that the rest of the AI-Capstone-Threat-Intel system can use. Because public threat intelligence appears in many formats, the Feed Collector must be flexible. A vendor advisory may appear as an HTML page, a government alert may be published through an RSS feed, a vulnerability list may be available as JSON, and a malware indicator list may be a plain text file. The collector should use source-specific adapters when necessary, but the output should follow a common record format.

The first step in ingestion is source registration. Each source should have a name, type, URL or endpoint, collection frequency, reliability notes, and parsing method. A source registry prevents the workflow from becoming a collection of disconnected one-off scrapers. It also makes it easier to disable a broken source without deleting the rest of the workflow. For example, a source entry might say that a vendor advisory feed is collected daily, parsed from RSS, and expected to provide a title, publication date, link, and summary.

After registration, the collector retrieves the content. The retrieval step should capture the HTTP status code, response time, retrieval timestamp, and any error message. If the source returns a successful response, the system should store either the raw content or enough of the original content to support future review. If the source fails, the collector should create an error record rather than silently skipping the source. Silent failure is dangerous in threat intelligence because the user may assume that no threats were found when the real issue was that collection failed.

Normalization converts inconsistent source formats into a shared structure. Typical normalized fields include source_name, source_url, source_type, collected_at, published_at, item_title, item_url, raw_text, extracted_entities, content_hash, and collection_status. The raw_text field should preserve the main body of the collected item. The extracted_entities field may contain lists of CVEs, IP addresses, domains, URLs, product names, malware names, threat actors, and ATT&CK technique references. Normalization should not remove the original source context. It should make the record easier to search while keeping enough evidence to verify the extraction.

Deduplication is important because multiple public sources often repeat the same advisory or indicator. The collector should use several deduplication signals instead of only one. A content hash can identify exact duplicates. A normalized URL can identify the same page collected more than once. A combination of CVE identifier, affected product, and publication date can identify records that discuss the same vulnerability. Indicator records can be deduplicated using the indicator value, indicator type, source, and first_seen timestamp.

The collector should distinguish exact duplicates from related records. If two sources mention the same CVE, that does not always mean one should be deleted. Multiple sources can strengthen confidence or provide different context. A better approach is to link related records under a shared entity while preserving the original source records. For example, two advisories about the same CVE may be merged at the entity level but still retained as separate evidence entries.

Data quality checks should run after normalization and deduplication. The system should flag records with missing titles, missing source URLs, invalid CVE formats, empty extracted text, or suspiciously short content. Quality flags do not always mean that the record is useless, but they help downstream components avoid over-trusting weak data. A strong Feed Collector produces records that are consistent, reviewable, and ready for retrieval or scoring.
